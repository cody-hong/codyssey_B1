# 작업 과정 정리

## 1. 문서 목적

이 문서는 Codyssey B1-1 미션을 완성하기까지 어떤 대화를 했고, 어떤 판단을 내렸으며, 산출물이 어떻게 발전했는지를 정리한 기록이다.

정리 목적은 다음과 같다.

- 어떤 방식으로 미션 방향을 잡았는지 남긴다.
- 프롬프트가 어떤 과정을 거쳐 개선되었는지 정리한다.
- 실제로 어떤 파일을 만들고 수정했는지 추적한다.
- 최종적으로 무엇이 완성되었는지 한눈에 확인한다.

## 2. 미션 개요

- 미션명: Codyssey B1-1, ChatGPT에게 일을 제대로 시키는 법 배우기
- 선택한 과업명: Codyssey 미션 코치 (교육 미션 분석, 개념 설명, 자기주도학습 계획 생성 자동화)
- 최종 목표: 교육 프로그램 미션 안내문을 입력하면 AI가 요구사항 분석, 핵심 개념 설명, 제출 계획, 체크리스트, 검증 포인트를 정리해주는 프롬프트 패키지를 완성하는 것
- 필수 산출물 3개:
  - LLM 모델 비교 선정 보고서
  - 시스템 설계 문서
  - 실행 로그
- 보너스 과제:
  - 보너스 1: `codybot` 스킬 배포
  - 보너스 2: 미션 수행 로드맵 시각 자료 제작

## 3. 시작 상태

처음 상태는 다음과 같았다.

- AI에 대한 기본 지식은 어느 정도 있었지만, 단계별로 미션을 분석하고 학습하는 경험은 많지 않았다.
- 미션 안내문이 길고 요구사항이 많아 어디서부터 시작해야 할지 막막한 상태였다.
- 단순히 답을 받는 것이 아니라, 통과 가능한 제출물 구조를 처음부터 잡아야 한다는 필요가 있었다.

- 내가 한 말 또는 요청:

- 받은 답변 또는 결정:
  - 미션의 핵심은 여러 AI를 활용해 좋은 프롬프트를 만들고, 결과를 비교하고, 개선 과정을 문서화하는 데 있다는 점을 확인했다.
- 그 결과 바뀐 점:
  - 단순 사용이 아니라 비교, 검증, 개선이 핵심이라는 방향이 잡혔다.

### 2단계. 과업명 확정하기

- 내가 한 말 또는 요청:
  - 실제 상황과 연결되는 과업명을 정리하고 싶다고 했다.
- 받은 답변 또는 결정:
  - 최종 과업명을 `Codyssey 미션 코치 (교육 미션 분석, 개념 설명, 자기주도학습 계획 생성 자동화)`로 확정했다.
- 그 결과 바뀐 점:
  - 이후 모든 문서가 같은 과업명을 기준으로 정리되었다.

### 3단계. 모델 비교 전략 세우기

- 내가 한 말 또는 요청:
  - ChatGPT, Claude, Gemini를 모두 활용하고, 무료/유료까지 포함해 6개 환경으로 확장하고 싶다고 했다.
- 받은 답변 또는 결정:
  - 모델 비교는 6개 환경으로 진행하되, 실제 B1-1 원문 대신 가상 미션을 동일 입력 데이터로 사용하기로 결정했다.
- 그 결과 바뀐 점:
  - `01. LLM 모델 비교 선정 보고서.md`의 실험 범위가 명확해졌다.

### 4단계. v1 프롬프트 만들기

- 내가 한 말 또는 요청:
  - 사람이 처음 생각나는 대로 말하는 수준의 v1 프롬프트를 만들고 싶다고 했다.
- 받은 답변 또는 결정:
  - 간단한 자연어 프롬프트를 기준선으로 삼고, 이후 개선 방향을 비교하는 구조로 가기로 했다.
- 그 결과 바뀐 점:
  - v1은 일부러 덜 정교하게 만들어 기본 성능 비교 기준으로 사용되었다.

### 5단계. 각 AI 응답 수집 및 비교

- 내가 한 말 또는 요청:
  - GPT 무료, Gemini 무료, Claude 무료, 이후 유료 응답들을 차례대로 공유했다.
- 받은 답변 또는 결정:
  - 각 응답을 요구사항 충족도, 구조화, 개념 설명, 확인 질문, 보너스 구분, 실용성 관점에서 비교했다.
- 그 결과 바뀐 점:
  - 최종 선정 모델을 ChatGPT 유료(GPT 5.5 Pro)로 정리할 수 있었다.

### 6단계. v2, v3로 프롬프트 개선

- 내가 한 말 또는 요청:
  - 더 좋은 구조, 왕초보 대응, 섹션별 진행, 참고자료 추천, 한국어 유튜브 기준 등을 점점 추가했다.
- 받은 답변 또는 결정:
  - v2에서는 구조화, Few-shot, 환각 방지, 확인 질문 규칙을 강화했고, v3에서는 왕초보 단계, 섹션형 진행, 참고자료 추천 규칙을 추가했다.
- 그 결과 바뀐 점:
  - 최종 프롬프트가 단순 분석기가 아니라 학습 코치형 구조로 발전했다.

### 7단계. 실행 로그와 검증 기준 정리

- 내가 한 말 또는 요청:
  - 10턴 이상 로그를 어떤 방식으로 남기면 되는지 질문하고, 실제 피드백을 계속 추가했다.
- 받은 답변 또는 결정:
  - 원본 대화와 요약 문서를 분리하고, 조건 변경, 추가 정보, 문맥 유지, 수정 포인트를 함께 기록하기로 했다.
- 그 결과 바뀐 점:
  - `03. 실행 로그.md`와 `04. 실행로그 전문.md`가 역할별로 분리되었다.

### 8단계. 보너스 과제 반영

- 내가 한 말 또는 요청:
  - `codybot` 스킬 배포 구조와 로드맵 이미지를 보너스 과제로 연결하고 싶다고 했다.
- 받은 답변 또는 결정:

- 그 결과 바뀐 점:
  - README와 시스템 설계 문서에 보너스 과제가 명확히 반영되었다.

### 9단계. GitHub 제출 구조 정리

- 내가 한 말 또는 요청:
  - GitHub 저장소를 만들고, 불필요한 파일은 빼고, 제출용 구조를 정리하고 싶다고 했다.
- 받은 답변 또는 결정:
  - 제출용 문서와 보너스 산출물만 GitHub에 남기고, 임시 테스트 파일은 추적에서 제외하기로 했다.
- 그 결과 바뀐 점:
  - 제출자가 보게 될 저장소 구조가 훨씬 깔끔해졌다.

## 5. 주요 의사결정 기록

- 실제 B1-1 원문 대신 가상 미션으로 모델 비교를 진행하기로 했다.
- 비교 대상은 3개 제품군, 총 6개 환경으로 확장했다.
- 최종 과업명은 `Codyssey 미션 코치 (교육 미션 분석, 개념 설명, 자기주도학습 계획 생성 자동화)`로 확정했다.
- 학습 레벨은 왕초보, 초보자, 중급자, 전문가 4단계 구조로 정리했다.
- 하위 프롬프트를 직접 입력하는 방식보다 메인 프롬프트 중심 구조로 정리했다.
- 보너스 1은 `codybot` 스킬 배포 구조로 연결했다.
- 보너스 2는 로드맵 이미지로 연결했다.
- 왕초보 모드는 과하게 확장하지 않고, 기본 단어 설명을 추가하는 수준으로 정리했다.
- 유튜브 추천은 한국어 기준과 검증 규칙을 둔 형태로 보강했다.

## 6. 프롬프트 발전 과정

### v1

- 왜 만들었는지:
  - 사용자가 처음 AI에게 도움을 요청하는 자연스러운 수준의 기준선을 만들기 위해
- 한계:
  - 구조가 약하고, 필수/보너스 구분이 흐릴 수 있었으며, 확인 질문이 거의 없었다.
- 다음 버전에서 고친 점:
  - 출력 구조, 역할, 검증 기준, 확인 질문, Few-shot 방향을 추가했다.

### v2

- 왜 만들었는지:
  - v1 비교 결과에서 드러난 누락과 모호함을 줄이기 위해
- 한계:
  - 구조는 좋아졌지만 왕초보 대응과 단계별 진행성이 부족했다.
- 다음 버전에서 고친 점:
  - 학습 레벨 분화, 왕초보 단계, 섹션별 진행, 참고자료 규칙을 추가했다.

### v3

- 왜 만들었는지:
  - 재사용성과 실제 사용 흐름을 더 좋게 만들기 위해
- 한계:
  - 처음에는 유튜브 추천 기능이 약했고, 왕초보 모드가 다소 무거워질 수 있었다.
- 다음 보강에서 고친 점:
  - 한국어 유튜브 기준을 추가하고, 왕초보는 기본 단어 설명 중심으로 가볍게 조정했다.

### v3 보강

- 보강 1:
  - 한국어 유튜브 및 참고자료 검증 기준 추가
- 보강 2:
  - AI, 프롬프트, 미션, 제출물, 요구사항, 체크리스트 같은 기본 단어 설명 강화

## 7. 산출물 제작 과정

### 01. 모델 비교 보고서

- 6개 AI 환경 응답을 수집했다.
- 평가 축을 만들고 점수표를 작성했다.
- 최종 선정 모델과 선정 근거를 정리했다.

### 02. 시스템 설계 문서

- 문제 정의, 타겟 사용자, 입력 템플릿을 정리했다.
- 페르소나, 시스템 프롬프트, 최종 유저 프롬프트를 정리했다.
- Few-shot 예시와 환각 검증 질문을 넣었다.
- 보너스 1, 보너스 2 반영 섹션을 추가했다.

### 03. 실행 로그

- 10턴 이상 대화 흐름을 요약했다.
- 조건 변경, 추가 정보, 문맥 유지 검증을 함께 정리했다.
- 프롬프트가 개선된 이유를 Turn 단위로 추적했다.

### 04. 실행로그 전문

- 원본 대화를 그대로 보관했다.
- 요약 문서에서 설명한 Turn 1~14의 근거 기록 역할을 하게 했다.

### README

- 저장소 소개, 산출물 위치, 보너스 과제, 요구사항 충족 체크를 정리했다.
- 평가자가 처음 봐도 구조를 이해할 수 있게 제출용 안내 문서로 정리했다.

## 8. 실제 작업 로그

- 미션 원문 텍스트를 마크다운 파일로 정리했다.
- 문서 제목 규칙을 통일했다.
- 모델 비교 보고서, 시스템 설계 문서, 실행 로그 문서를 작성했다.
- 실행 로그 전문 파일을 별도로 만들었다.
- handoff 문서와 AGENTS 문서를 정리했다.
- 최종 프롬프트 테스트 파일을 만들었다가, 제출 구조에서는 제외되도록 정리했다.
- Git 저장소를 초기화하고 GitHub에 연결했다.
- `codybot` 스킬 구조를 추가했다.
- `.claude-plugin` 메타데이터를 추가했다.
- `gemini-extension.json`, `GEMINI.md`, `commands/codybot.toml`을 추가했다.
- 보너스 2용 로드맵 이미지를 `assets/`에 추가했다.

## 9. 어려웠던 점과 해결 방법

### 요구사항이 많아서 핵심이 흐려졌던 문제

- 어려웠던 점:
  - 미션 원문이 길고 요구사항이 많아 중심을 잡기 어려웠다.
- 해결 방법:
  - 체크리스트 구조와 산출물 중심 정리 방식으로 풀었다.

### 보너스가 필수처럼 보일 수 있었던 문제

- 어려웠던 점:
  - 일부 응답은 보너스 과제를 필수처럼 섞어 보여줄 수 있었다.
- 해결 방법:
  - 필수와 선택을 문서와 프롬프트 모두에서 분리했다.

### 왕초보 모드가 너무 무거워질 뻔한 문제

- 어려웠던 점:
  - 왕초보 대응을 강화하다 보면 설명이 과하게 길어질 수 있었다.
- 해결 방법:
  - 전체 구조를 바꾸지 않고, 기본 단어 설명만 추가하는 방향으로 정리했다.

### 유튜브 추천이 약했던 문제

- 어려웠던 점:
  - 모델이 링크 환각을 피하려다 보니 추천을 거의 하지 않는 경우가 있었다.
- 해결 방법:
  - 한국어 기준, 검증된 채널, 검색 키워드 제공 규칙을 추가했다.

### GitHub push 인증 문제

- 어려웠던 점:
  - 처음 push할 때 GitHub 인증 문제가 있었다.
- 해결 방법:
  - Personal Access Token 방식으로 인증해 push를 완료했다.

이번 과정을 통해 중요한 것은 단순히 AI 답변을 받는 것이 아니라, 여러 모델을 비교하고, 결과를 검증하고, 반복적으로 수정하면서 최종적으로 재사용 가능한 형태로 정리하는 것이라는 점을 배웠다.
