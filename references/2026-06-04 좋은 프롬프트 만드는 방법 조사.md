# 좋은 프롬프트 만드는 방법 조사

작성일: 2026-06-04

## 조사 목적

Codyssey B1-1 미션의 핵심은 3개 이상의 LLM을 비교하고, 특정 업무 과업에 적합한 프롬프트를 설계·검증·개선하는 것입니다. 이 문서는 최신 공식 문서와 신뢰도 높은 AI 엔지니어링 자료를 바탕으로, 좋은 프롬프트를 만드는 방법, 구조, 예시, 평가 방법을 정리합니다.

## 조사 기준

- 최신성: 2026-06-04 기준 최근 약 6개월 이내 자료를 우선 사용했습니다.
- 신뢰도: OpenAI, Codex, Anthropic/Claude, Google Cloud, Microsoft Learn, NIST, Braintrust 등 공식 문서 또는 인지도 높은 AI 엔지니어링 팀 자료를 우선했습니다.
- 적용성: B1-1 미션 산출물인 모델 비교 보고서, 시스템 설계 문서, 실행 로그에 바로 활용 가능한 내용만 선별했습니다.

## 한 줄 결론

좋은 프롬프트는 멋진 문장이 아니라, **목표·역할·입력·출력 형식·제약·예시·검증 기준을 명확히 정의한 업무 지시서**에 가깝다. 최신 LLM에서는 “마법의 문구”보다 구조화, 근거 제시, 실패 기준, 반복 평가가 더 중요하다.

## 핵심 원칙

### 1. 먼저 성공 기준을 정합니다

프롬프트를 쓰기 전에 “좋은 결과”가 무엇인지 정의해야 합니다. OpenAI와 Anthropic 모두 프롬프트 개선보다 먼저 성공 기준과 평가 방법을 정하라고 권합니다.

좋은 성공 기준 예시:

- 제공된 회의록 안에 있는 사실만 사용합니다.
- 결정사항, Action Items, 리스크, 확인 필요 항목을 분리합니다.
- 담당자 실명 대신 역할명만 사용합니다.
- 추측이 필요한 내용은 “확인 필요”로 표시합니다.
- 출력은 Markdown 표 또는 불릿 형식을 따른다.

나쁜 성공 기준 예시:

- 잘 요약해줘.
- 자연스럽게 써줘.
- 보기 좋게 정리해줘.

### 2. 역할보다 업무 목적이 먼저다

페르소나는 유용하지만 과장된 역할만으로는 부족합니다. “세계 최고의 전문가처럼 행동해”보다 “프로젝트 매니저 관점에서 회의록을 결정사항과 후속 작업 중심으로 정리해”가 더 좋습니다.

좋은 역할 정의는 다음을 포함합니다.

- 역할: 어떤 관점으로 답할 것인가
- 목표: 어떤 업무 결과를 만들 것인가
- 우선순위: 정확성, 간결성, 친절함, 형식 준수 중 무엇을 우선할 것인가
- 금지 사항: 추측, 개인정보 노출, 근거 없는 단정 등

### 3. 지시와 데이터를 분리합니다

OpenAI와 Anthropic은 Markdown 제목, XML 태그, 구분자 등을 사용해 프롬프트의 영역을 나누라고 권합니다. 모델이 “무엇이 지시이고, 무엇이 입력 데이터인지” 구분하기 쉬워진다.

권장 구조:

```text
# 역할
너는 [역할]이다.

# 목표
[해야 할 업무]를 수행한다.

# 규칙
- [출력 규칙]
- [금지 사항]
- [확인 질문 규칙]
- [근거/불확실성 처리 규칙]

# 입력
<input>
[사용자 입력 데이터]
</input>

# 출력 형식
[원하는 형식]
```

### 4. 출력 형식을 구체적으로 고정합니다

“짧게”, “자세히”, “깔끔하게” 같은 표현은 사람마다 기준이 다르다. 최신 문서들은 길이, 형식, 섹션, 표 여부, 불릿 수처럼 확인 가능한 조건을 쓰라고 권합니다.

나쁜 예:

```text
회의록을 깔끔하게 요약해줘.
```

좋은 예:

```text
아래 회의록을 다음 형식으로 요약해줘.

1. 결정사항: 최대 5개
2. Action Items: 담당 역할 / 작업 / 기한 / 근거 문장
3. 리스크: 확정된 리스크와 추정 리스크를 분리
4. 확인 필요: 정보가 부족한 항목

회의록에 없는 내용은 추측하지 말고 "확인 필요"로 표시해.
```

### 5. Few-shot 예시는 실제 실패를 막는 용도로 쓴다

Few-shot은 모델에게 “좋은 입력 → 좋은 출력” 패턴을 보여주는 방식입니다. Anthropic은 3-5개의 관련성 높고 다양한 예시를 권장합니다. B1-1 미션에서는 최소 3개가 필요합니다.

좋은 Few-shot 예시 조건:

- 실제 과업과 유사하다.
- 정상 입력, 애매한 입력, 엣지 케이스를 포함합니다.
- 출력 형식이 최종 프롬프트와 동일하다.
- 모호한 입력에서는 AI가 질문해야 하는 지점을 보여준다.

### 6. 단계적 접근은 “내부 검토 + 최종 요약”으로 설계합니다

B1-1 미션에는 단계적 추론 유도 전/후 비교가 필요합니다. 다만 최신 OpenAI 문서는 reasoning 모델에는 “생각을 단계별로 모두 출력해”가 불필요하거나 방해될 수 있다고 설명합니다. 따라서 최종 제출용 프롬프트에서는 장문의 사고 과정을 노출시키기보다, 내부적으로 점검하고 최종 답변만 간결히 내게 하는 방식이 안전하다.

권장 표현:

```text
내부적으로 다음 순서로 검토해.
1. 입력 정보에서 확정 사실과 불확실한 정보를 구분한다.
2. 출력 형식에 필요한 항목이 누락되었는지 확인한다.
3. 누락된 정보가 있으면 최대 3개의 확인 질문을 한다.
4. 충분한 정보가 있으면 최종 결과만 출력한다.

최종 답변에는 장문의 추론 과정을 쓰지 말고, 핵심 근거만 짧게 표시한다.
```

### 7. 환각 방지 규칙을 명시합니다

환각은 사실, 수치, 정책, 계산처럼 검증 가능한 내용에서 근거 없이 틀린 정보를 확신하는 답변입니다. Anthropic은 모델에게 “모르면 모른다고 말해도 된다”는 규칙과 근거 기반 응답을 명시하라고 권합니다.

환각 방지 규칙 예시:

```text
사실, 정책, 수치, 계산이 포함된 답변은 반드시 다음 중 하나를 따른다.
- 제공된 입력 안에 근거가 있으면 해당 근거를 짧게 표시한다.
- 근거가 없으면 "확인 필요"라고 표시하고, 무엇을 확인해야 하는지 말한다.
- 알 수 없는 내용을 확정적으로 말하지 않는다.
```

## 추천 프롬프트 구조

아래 구조를 B1-1 미션의 시스템 프롬프트 기본형으로 사용할 수 있습니다.

```text
# Identity
너는 [업무 분야]를 돕는 [역할/페르소나]다.
너의 목표는 [타겟 사용자]가 [업무 문제]를 빠르고 정확하게 해결하도록 돕는 것이다.

# Priority
1. 정확성
2. 입력 근거 준수
3. 출력 형식 준수
4. 간결하고 실무적인 표현

# Task
사용자가 제공한 [입력 데이터]를 바탕으로 [업무 결과물]을 작성한다.

# Rules
- 입력에 없는 사실은 만들지 않는다.
- 정보가 부족하면 최대 3개의 확인 질문을 먼저 한다.
- 개인정보는 실명 대신 역할명으로 바꾼다.
- 사실/정책/수치/계산은 근거가 없으면 "확인 필요"로 표시한다.
- 최종 답변에는 불필요한 장문의 추론 과정을 노출하지 않는다.

# Output Format
다음 형식을 반드시 따른다.
1. [섹션 A]
2. [섹션 B]
3. [섹션 C]
4. 확인 필요

# Examples
<examples>
  <example id="1">
    <input>[좋은 입력 예시]</input>
    <output>[좋은 출력 예시]</output>
  </example>
  <example id="2">
    <input>[모호한 입력 예시]</input>
    <output>[확인 질문 예시]</output>
  </example>
  <example id="3">
    <input>[엣지 케이스 입력 예시]</input>
    <output>[안전한 출력 예시]</output>
  </example>
</examples>

# User Input
<input>
{{사용자 입력}}
</input>
```

## 업무 과업별 예시

### 예시 1: 회의록 요약 프롬프트

```text
너는 프로젝트 매니저 출신 업무 자동화 코치다.
사용자가 제공한 회의 메모를 사내 공유용 회의록으로 정리한다.

규칙:
- 회의 메모에 없는 내용은 추측하지 않는다.
- 담당자 실명은 역할명으로 바꾼다.
- 결정사항과 Action Items를 분리한다.
- 기한이나 담당자가 불명확하면 "확인 필요"로 표시한다.
- 최종 답변에는 장문의 추론 과정을 쓰지 않는다.

출력 형식:
1. 회의 요약: 3문장 이내
2. 결정사항: 불릿
3. Action Items: 표(담당 역할 / 작업 / 기한 / 근거)
4. 리스크: 확정 / 추정 구분
5. 확인 필요

회의 메모:
"""
{{회의 메모}}
"""
```

### 예시 2: 메일 초안 작성 프롬프트

```text
너는 B2B 업무 커뮤니케이션을 돕는 비즈니스 메일 작성 코치다.
사용자의 상황과 목적을 바탕으로 상대방에게 보낼 메일 초안을 작성한다.

규칙:
- 요청 목적을 첫 문단에서 분명히 밝힌다.
- 상대방을 압박하지 않되, 필요한 행동은 명확히 요청한다.
- 날짜, 금액, 정책 등 사실 정보가 부족하면 임의로 만들지 않는다.
- 정보가 부족하면 초안 작성 전에 최대 3개의 확인 질문을 한다.

입력 템플릿:
[목적]
[대상]
[상황]
[필수 포함 항목]
[톤]
[금지 표현]

출력 형식:
- 제목 후보 2개
- 메일 본문
- 확인 필요
```

### 예시 3: 요구사항 정리 프롬프트

```text
너는 서비스 기획자와 개발팀 사이에서 요구사항을 정리하는 프로덕트 분석가다.
사용자의 설명을 사용자 스토리와 수용 기준으로 변환한다.

규칙:
- 기능 요구사항과 비기능 요구사항을 분리한다.
- 구현 방식은 확정하지 말고 요구사항 수준으로 정리한다.
- 애매한 범위는 확인 질문으로 남긴다.
- 개인정보나 민감 정보는 일반화한다.

출력 형식:
1. 문제 정의
2. 타겟 사용자
3. 사용자 스토리
4. 수용 기준
5. 예외/엣지 케이스
6. 확인 필요
```

## Few-shot 예시 샘플

아래는 회의록 요약 과업 기준 샘플입니다.

### Example 1: 정상 입력

입력:

```text
마케팅팀은 6월 캠페인 랜딩페이지를 6월 14일까지 완성하기로 했다.
디자인팀은 6월 7일까지 시안을 공유하고, 개발팀은 6월 12일까지 퍼블리싱한다.
예산은 아직 확정되지 않았다.
```

출력:

```text
1. 회의 요약
- 6월 캠페인 랜딩페이지 제작 일정이 논의되었다.
- 디자인과 개발 일정은 확정되었으나 예산은 미확정이다.

2. 결정사항
- 랜딩페이지는 6월 14일까지 완성한다.

3. Action Items
| 담당 역할 | 작업 | 기한 | 근거 |
| --- | --- | --- | --- |
| 디자인팀 | 시안 공유 | 6월 7일 | "디자인팀은 6월 7일까지 시안을 공유" |
| 개발팀 | 퍼블리싱 | 6월 12일 | "개발팀은 6월 12일까지 퍼블리싱" |

4. 리스크
- 확정 리스크: 예산 미확정

5. 확인 필요
- 최종 예산 규모
```

### Example 2: 모호한 입력

입력:

```text
다음 주까지 다 정리하기로 했고, 담당자는 그때 얘기한 사람으로 하면 돼.
```

출력:

```text
정보가 부족해 회의록을 확정하기 어렵습니다. 아래 내용을 확인해 주세요.

1. "다음 주"의 정확한 날짜 또는 마감일은 언제인가요?
2. "다 정리"가 의미하는 작업 범위는 무엇인가요?
3. "그때 얘기한 사람"의 역할명은 무엇인가요?
```

### Example 3: 추측 위험 입력

입력:

```text
신규 요금제는 아마 법무 검토가 끝났을 거야. 다음 공지에 넣자.
```

출력:

```text
1. 회의 요약
- 신규 요금제를 다음 공지에 포함하는 방안이 언급되었다.
- 법무 검토 완료 여부는 확정되지 않았다.

2. 결정사항
- 확인된 결정사항 없음

3. Action Items
| 담당 역할 | 작업 | 기한 | 근거 |
| --- | --- | --- | --- |
| 담당 부서 | 법무 검토 완료 여부 확인 | 확인 필요 | "아마 법무 검토가 끝났을 거야" |

4. 리스크
- 추정 리스크: 법무 검토가 완료되지 않았을 가능성

5. 확인 필요
- 법무 검토 완료 여부
- 공지 반영 가능 시점
```

## 모델 비교에 사용할 평가 축

B1-1 미션에서는 3개 이상의 LLM에 동일한 과업을 수행시키고 점수와 근거를 남겨야 합니다. 다음 평가 축을 추천합니다.

| 평가 축 | 설명 | 점수 기준 |
| --- | --- | --- |
| 정확성 | 입력에 있는 사실만 사용했는가 | 1=환각 많음, 5=근거 기반 |
| 형식 준수 | 요청한 출력 형식을 지켰는가 | 1=형식 무시, 5=완전 준수 |
| 한국어 자연스러움 | 실무 문서로 읽기 좋은가 | 1=어색함, 5=자연스러움 |
| 확인 질문 능력 | 정보 부족 시 적절히 되물었는가 | 1=추측, 5=핵심 질문 |
| 문맥 유지 | 10턴 대화에서 조건을 유지했는가 | 1=자주 잊음, 5=일관 유지 |
| 안전성/환각 대응 | 모르는 내용을 단정하지 않았는가 | 1=단정, 5=확인 필요 표시 |
| 속도/비용 | 무료/유료, 응답 속도, 사용 제한 | 1=부담 큼, 5=실용적 |

## 프롬프트 개선 절차

### v1: 간단 프롬프트

```text
아래 회의 내용을 요약해줘.

{{회의 메모}}
```

예상 문제:

- 출력 형식이 매번 달라질 수 있습니다.
- 담당자, 기한, 리스크가 빠질 수 있습니다.
- 불확실한 내용을 사실처럼 말할 수 있습니다.
- 정보가 부족해도 확인 질문 없이 추측할 수 있습니다.

### v2: 개선 프롬프트

```text
너는 프로젝트 매니저 출신 업무 자동화 코치다.
아래 회의 메모를 사내 공유용 회의록으로 정리한다.

내부적으로 다음을 검토한다.
1. 확정된 사실과 불확실한 내용을 구분한다.
2. 결정사항, Action Items, 리스크를 분리한다.
3. 담당자/기한/근거가 누락된 항목은 확인 필요로 표시한다.

규칙:
- 입력에 없는 사실은 만들지 않는다.
- 담당자 실명은 역할명으로 바꾼다.
- 법무/정책/수치 관련 내용은 근거가 없으면 확정하지 않는다.
- 최종 답변에는 장문의 추론 과정을 쓰지 않는다.

출력 형식:
1. 회의 요약
2. 결정사항
3. Action Items: 표
4. 리스크
5. 확인 필요

회의 메모:
"""
{{회의 메모}}
"""
```

개선 포인트:

- 역할과 목적이 명확하다.
- 출력 형식이 고정되어 비교 평가가 쉽습니다.
- 환각 방지 규칙이 들어 있습니다.
- 확인 질문과 확인 필요 항목을 분리할 수 있습니다.

## 환각 검증 질문 설계

환각 검증은 “정답을 맞혔는가”뿐 아니라 “모르면 모른다고 했는가”까지 평가해야 합니다.

| 질문 | 기대 동작 | Pass 기준 | Fail 기준 |
| --- | --- | --- | --- |
| 제공된 회의록에 예산 금액이 있나요? | 없으면 없다고 말함 | 근거 없음/확인 필요 표시 | 금액을 지어냄 |
| 법무 검토가 완료됐나요? | 입력 근거 확인 | 완료 여부 불명확 표시 | 완료됐다고 단정 |
| Action Item 담당자는 누구인가요? | 역할명 기준 정리 | 담당 역할 또는 확인 필요 | 임의의 이름 생성 |
| 다음 회의 날짜는 언제인가요? | 입력에 없으면 확인 질문 | 날짜 확인 필요 표시 | 임의 날짜 제시 |
| 정책상 공지 가능한가요? | 정책 자료 없음을 밝힘 | 정책 문서 확인 필요 | 가능/불가능 단정 |

## 10턴 대화 로그 설계 팁

10턴 로그는 단순 대화량 채우기가 아니라 문맥 유지 검증용입니다. 다음 흐름이 좋습니다.

1. 사용자: 회의 메모 제공
2. AI: 확인 질문 2-3개
3. 사용자: 일부 정보 제공
4. AI: 1차 회의록 생성
5. 사용자: 톤을 더 간결하게 변경
6. AI: 톤만 변경하고 기존 사실 유지
7. 사용자: 담당자 실명을 역할명으로 바꾸라고 요청
8. AI: 역할명 변경, 나머지 유지
9. 사용자: 법무 검토가 불확실하다고 추가 정보 제공
10. AI: 법무 관련 내용을 확인 필요로 수정

Pass 기준:

- 처음 정한 출력 형식이 유지됩니다.
- 중간 조건 변경은 변경된 부분만 반영합니다.
- 이전에 확정된 사실을 잊지 않습니다.
- 불확실한 내용은 끝까지 “확인 필요”로 둔다.

## B1-1 미션에 추가하면 좋은 부분

사용자가 물은 “더 필요한 부분”에 대한 답은 다음과 같습니다. 필수 요구사항 외에 아래를 넣으면 보고서의 완성도가 좋아진다.

- 프롬프트 버전 표: v1, v2, 변경 이유, 개선 결과를 한 표로 정리
- 테스트 케이스 표: 정상 입력, 모호한 입력, 환각 위험 입력, 형식 준수 입력
- 모델 사용 환경 표: 모델명, 웹/앱/API, 무료/유료, 사용 날짜, 주요 설정
- 평가 루브릭: 각 평가 축의 1점/3점/5점 기준
- 실패 사례 기록: 어떤 모델이 어떤 실수를 했고, 프롬프트를 어떻게 고쳤는지
- 최종 프롬프트 전문: 제출자가 실제로 재사용할 수 있는 형태
- 원본 로그와 편집본 분리: 원본 대화 전문 + 읽기 쉽게 정리한 요약본

## 최종 체크리스트

좋은 프롬프트인지 확인하려면 아래 질문에 답해본다.

- 이 프롬프트의 업무 목적이 한 문장으로 명확한가?
- 타겟 사용자와 입력 데이터 형태가 드러나는가?
- 출력 형식이 평가 가능할 정도로 구체적인가?
- 입력에 없는 사실을 만들지 말라는 규칙이 있는가?
- 정보가 부족할 때 확인 질문을 하도록 되어 있는가?
- Few-shot 예시가 최소 3개 있고, 모호한 입력 예시가 포함되어 있는가?
- v1과 v2의 차이가 분명하고 개선 이유가 설명 가능한가?
- 환각 검증 질문 5개 이상으로 테스트할 수 있는가?
- 10턴 이상 대화에서 문맥 유지 여부를 확인할 수 있는가?
